# 自动调度

基于优先级、依赖关系和并行开发限制，自动调度并启动特性。

## 参数

- `--dry-run`: 显示将要调度的内容而不实际启动
- `--max=<n>`: 此次运行覆盖 max_concurrent
- `--force`: 即使达到并行限制也启动特性
- `--priority-threshold=<n>`: 只启动超过此优先级的特性

## 第 1 步：加载配置

读取 `feature-workflow/config.yaml`：
- `workflow.max_concurrent`
- `workflow.auto_start`
- 其他相关设置

## 第 2 步：分析待处理特性

对于每个待处理特性：
- 检查依赖是否满足
- 检查优先级阈值
- 计算优先级分数
- 识别父/子关系

## 第 3 步：确定容量

计算可用容量：
```python
active_count = len(queue.yaml['active'])
max_concurrent = config['workflow']['max_concurrent']
available = max_concurrent - active_count
```

## 第 4 步：选择要启动的特性

按以下排序待处理特性：
1. 优先级（高到低）
2. 创建日期（较早的优先）
3. 依赖顺序（先决条件优先）

选择前 N 个特性，其中 N = 可用容量

## 第 5 步：验证选择

对于每个选定的特性：
- 验证依赖已完成
- 检查父/子约束
- 验证无冲突

## 第 6 步：执行或预览

如果 `--dry-run`：
- 显示将要启动的内容
- 显示预期完成顺序
- 显示资源利用率

如果不是 `--dry-run`：
- 为每个选定的特性调用 `start-feature`
- 更新 queue.yaml
- 报告结果

## 调度算法

```
分数 = 优先级 + 依赖奖励 + 时长奖励

其中:
  优先级 = 用户定义的优先级 (1-100)
  依赖奖励 = +10 如果解锁其他特性
  时长奖励 = +1 每周（最多 +5）
```

## 示例

```
# 预览将要调度的内容
/auto-schedule --dry-run

# 正常调度
/auto-schedule

# 覆盖 max_concurrent
/auto-schedule --max=5

# 只启动高优先级特性
/auto-schedule --priority-threshold=80

# 即使达到限制也强制启动
/auto-schedule --force
```

## 注意事项

1. **尊重依赖** - 从不启动有未满足依赖的特性
2. **基于优先级** - 高优先级特性先启动
3. **感知父/子** - 处理拆分特性关系
4. **感知资源** - 尊重 max_concurrent 限制
5. **安全模式** - 默认使用 --dry-run 预览
